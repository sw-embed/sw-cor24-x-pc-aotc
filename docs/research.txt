



I have a custom p-code VM implemented in COR24 ISA assembler.  I have a Pascal compiler (cross-compiled from C to the same assembler).  The VM is an interpeter, without a JIT (p-code to to machine code) compiler.  It might also be useful (if possible) to have an AOT (p-code to .s) translator/compiler.  How would I implement this?  The JIT would be running on the cor24 ISA, so it would have to be cross-compiled from C to .s and then assembled/linked into the VM.  The AOT tool could be written in Rust to take p-code and produce .s files (but it would be nice if the AOT tool could run on cor24 ISA directly, so this requires implementing it in C.  Discuss the feasibility and potential issues and come up with recommendations and a plan.

You have three related implementation choices:

Stay with the interpreter only

Add a JIT inside the COR24 VM/runtime

Add an AOT translator from p-code to COR24 .s

Given your environment, I would treat these as an evolutionary ladder, not parallel first efforts:

First: make the interpreter better and more compiler-friendly

Second: build an AOT p-code → .s translator

Third: only after that, consider a very small tracing/template JIT, if it still seems worth it

The reason is architectural: an AOT backend and a JIT backend share most of the same hard problems, but the AOT version is dramatically easier to debug, test, and bootstrap.

2. High-level design choice
Recommendation in one sentence
Implement AOT first, and structure it so that a future JIT can reuse the same opcode-lowering logic.

That means:

define a single internal lowering model

write one backend that emits .s

later, optionally write another backend that emits machine words into memory

This gives you a clean separation:

front end: decode p-code and build IR / basic-block info

middle end: analysis and lowering decisions

back ends:

AOT textual assembler emitter

JIT machine-code emitter

3. Feasibility of each option

Feasibility
Very high. You already have it.

Benefits

Risks / downsides

B. AOT p-code → COR24 .s
Feasibility
High and strongly recommended.

This is the best next step because:

you can diff it, test it, single-step it

Benefits

Risks / downsides
need to define a real mapping from VM semantics to machine ABI

may expose impedance mismatch between p-code stack semantics and COR24 register model

code size may grow

some VM operations may lower poorly without redesign

Recommendation
This should be Phase 1.

C. JIT on COR24 itself
Feasibility
Possible, but expensive and risky in your environment.

A JIT on a small custom integer-only architecture is not impossible, but it creates hard requirements:

writable/executable code memory model

code cache management

relocations/patching

callout interface between JITted code and runtime helpers

invalidation / cleanup strategy

more RAM pressure

debugging complexity goes way up

Since COR24 is resource-constrained and educational, the JIT likely gives less payoff per unit effort than AOT.

Benefits
can compile only hot code

can specialize based on runtime conditions

can keep interpreted fallback for rarely used code

interesting as a research/teaching project

Risks / downsides
runtime memory fragmentation

emitted-code bugs are much harder to diagnose

self-modifying/executable memory issues

generated code must obey exact calling conventions

exception/error handling gets more complex

likely needs a mini linker/relocator anyway

code generation in C on a tiny target is much harder than host-side Rust AOT

Recommendation
Do not start here.

4. Core architectural issue: what is your p-code model?
Before deciding JIT vs AOT, the biggest question is:

Is your p-code stack-based, register-like, or mixed?
This matters because translation difficulty depends heavily on VM shape.

If your p-code is stack-based
Example: LIT 1; LIT 2; ADD; STORE x

Then AOT/JIT can work, but naive translation may be poor:

lots of stack traffic

many push/pop sequences

harder optimization

code gets verbose

If your p-code has locals/frames plus implicit eval stack
This is common and still manageable.
You can:

map top-of-stack to registers

spill deeper stack values to memory

map locals to frame offsets

If your p-code is already closer to a 3-address form
Then translation gets much easier and better.

5. Recommendation: define a lowering-friendly VM contract
Before implementing any compiler backend, define these precisely.

A. Runtime memory model
Specify:

code segment

globals/static data

heap if any

stack(s):

Pascal call stack

VM eval stack, if separate

string/constant pool layout

B. Activation record format
Define:

return address

old frame pointer

local variables

temporaries

parameters

function result location

If interpreter and AOT/JIT use different frame layouts, life gets harder.
Best case: define one canonical runtime frame discipline and use it everywhere.

C. Calling convention
You need to decide:

which COR24 registers are caller-saved

which are callee-saved

where function results go

how runtime helpers are called

how parameters are passed

how VM/AOT/JIT interact

For example, you might reserve:

r0-r2 as scratch / expression registers

fp=r3

sp=r4

z=r5

iv=r6

ir=r7

That likely means AOT/JIT code will have only a few true working registers, so design for a register-starved backend.

D. Error/exception model
Define what happens for:

divide by zero

nil/invalid pointer

bounds error

illegal opcode

stack overflow

out-of-memory

This matters hugely for generated code.

6. Why AOT first is the right engineering move
AOT lets you solve the hard parts in a visible way
With AOT, when p-code block X becomes assembly Y, you can:

inspect generated .s

compare against hand-written .s

assemble and run it

single-step it in your existing debugger

keep the interpreter as oracle

write golden tests on exact emitted assembly patterns if useful

With JIT, when something goes wrong, you must debug:

the input p-code

the compiler logic

the binary emitter

relocation/patching

runtime state

generated code in RAM

That is a much worse first step.

7. Practical implementation strategies
Strategy 1: direct p-code → .s lowering
This is the simplest initial AOT design.

Pipeline

resolve labels / procedures / constants

split into basic blocks

compute stack depth at block boundaries

emit straightforward .s

Pros
simple

fast to implement

no extra IR required

good for proof of concept

Cons
optimization opportunities limited

stack discipline can get messy

JIT reuse later is weaker

Good use
Start here for a small subset.

Strategy 2: p-code → internal IR → .s
This is what I recommend once you move beyond proof of concept.

Internal IR should capture
loads/stores

arithmetic ops

branches

calls

frame accesses

constants

abstract stack slots or temporaries

The IR does not need to be fancy. It can be a simple linear three-address-ish form.

Pros
easier optimization

easier retargeting

much easier later JIT support

simplifies testing

Cons
more initial work

Recommendation
Do this after a minimal direct-lowering prototype proves viability.

8. AOT design options

9. JIT design options
If you later do a JIT, there are several levels.

Option 1: template/threaded “micro-JIT”
Each p-code opcode expands to a fixed template of machine code bytes.

Example
LIT k emits template to load constant

ADD emits template to combine TOS/NOS

LOAD local emits template for frame access

Benefits
easy compared to optimizing JIT

works well on small ISAs

close cousin of AOT direct lowering

Risks
code quality mediocre

still need code cache and patching

Recommendation
This is the only JIT style I would seriously consider early.

Option 2: tracing JIT
Compile hot loops only.

Benefits
potentially good payoff

avoids compiling cold code

Risks
trace recording and side exits are complex

not a great fit for early educational system unless you want that as the research topic

Recommendation
Later, much later.

Option 3: method/function JIT
Compile whole p-code procedures.

Benefits
conceptually simple

easier than tracing

good reuse of AOT backend

Risks
compile latency may be noticeable on COR24

need memory for compiled procedures

invalidation and caching still required

Recommendation
If you ever do JIT, this is probably better than tracing.

10. Key potential issues
10.1 Register starvation on COR24
You do not have abundant registers.

That means:

expression evaluation must be careful

stack-top caching should be small, perhaps 1–2 virtual stack slots in registers

many temporaries may spill

Recommendation
Design backend around:

TOS in one register when possible

NOS optionally in another

everything else in frame/eval stack memory

10.2 Code/data memory separation
Can COR24 safely:

write machine words into memory

then execute them

If yes, JIT is possible.
If not, JIT becomes much harder or impossible without special loader tricks.

Recommendation
Verify early:

executable RAM?

any alignment restrictions?

ability to branch/call into generated regions?

relocation assumptions?

10.3 Linking and symbol resolution
For AOT, .s can use your existing assembler/linker.

For JIT, you need some subset of a linker:

resolve addresses of runtime helpers

patch branches and calls

locate constants/string pools

Recommendation
This is another reason AOT first wins.

10.4 Pascal semantics that are awkward natively
Potential pain areas:

nested procedures / static links

sets

strings

variant records

file I/O abstractions

conformant arrays if you go that far

procedure/function variables

Recommendation
For first native backend, restrict supported features to:

integers

booleans

simple arrays

records

procedures/functions

no fancy strings/sets/files at first unless already runtime-backed cleanly

10.5 Debugging
Interpreter bugs are easier than generated-code bugs.

Recommendation
Build debug support into the AOT backend:

emit comments mapping p-code PC to .s

emit labels for procedures and blocks

optionally emit pseudo-source annotations

preserve a p-code-to-native address map

That map later becomes essential for JIT too.

10.6 Self-hosting pressure
You said it would be nice if the AOT tool could run on COR24 directly.

That is a good long-term goal, but it should not drive the first design too hard.

Recommendation
Implement in this order:

Rust host AOT

stabilize format and algorithms

reimplement a subset in C for COR24

optionally use same logic for JIT fragments

11. Best hybrid approach
The most practical architecture is:

Keep three execution modes
Mode 1: interpreted
reference semantics

debugging

fallback for unsupported features

Mode 2: AOT compiled
preferred for production/native execution

built on host first

later maybe available on COR24

Mode 3: optional JIT
only for experiments or hot code

maybe compile individual procedures

only after AOT is solid

This gives you graceful evolution.

16. Performance expectations
AOT
Likely gives the biggest practical speedup relative to effort, because it removes:

opcode dispatch overhead

repeated decode

some stack shuffling

JIT
Could outperform AOT only if it does meaningful runtime specialization, but on COR24:

memory is tighter

runtime compile cost matters more

optimization budget is small

So the likely ranking is:

Best effort/benefit: AOT

Best simplicity: interpreter

Best “interesting research project”: JIT

17. Plan I recommend
Stage 1 — formalize VM/runtime contracts
Write a short design doc defining:

p-code instruction set semantics

frame layout

calling convention

runtime helper ABI

memory regions

error behavior

Do this first. It will save enormous pain later.

Stage 2 — improve interpreter as reference oracle
Before codegen:

ensure interpreter is correct

add tracing mode:

stack state

locals/globals touched

add deterministic tests

You will use interpreter-vs-native differential testing.

Stage 3 — host-side Rust AOT prototype
Implement:

.s emitter

compile and link with runtime helpers

Target:

simple integer Pascal programs

This proves feasibility fast.

Stage 4 — differential test harness
For each program:

compare:

exit/result code

final globals if relevant

selected memory snapshots if useful

This is critical.

Stage 5 — add IR and better lowering
After prototype works:

insert a small IR

add basic block analysis

add stack-depth propagation

add trivial optimizations:

constant folding

dead temp elimination

top-of-stack caching

branch simplification

Stage 6 — broaden Pascal feature coverage
Add features gradually:

arrays

records

recursion

nested calls

parameter passing variants

heap/runtime ops

Do not try to do all Pascal features at once.

Stage 7 — optional C-hosted AOT on COR24
After host Rust version stabilizes:

port simplified backend to C

run on COR24

maybe emit .s text or a compact relocatable/native form

At that point you can decide whether emitting text .s on COR24 is actually worth it, or whether a compact binary/native object format is better.

Stage 8 — only then evaluate a JIT
Reuse as much lowering logic as possible.

Start with:

compile whole p-code procedures

template-based non-optimizing JIT

fallback to interpreter on failure

Do not begin with hot tracing or speculative optimization.

18. Additional recommendation: consider a binary AOT format before self-hosted .s
You said it would be nice if AOT could run directly on COR24 in C and produce .s.

That is understandable, but for a hosted-on-COR24 compiler, textual .s has costs:

large output size

assembler pass required

string formatting overhead

symbol management overhead

A better long-term self-hosted path may be:

p-code → native object-ish binary

then either:

load directly, or

pass through a minimal relocator/linker

So I would distinguish:

host Rust AOT: emit .s first, because it is easy to debug

future COR24 self-hosted AOT: probably emit a compact binary/native format, not text .s

That can still coexist with the .s backend for debugging.

19. Risks and mitigations
Risk: p-code too tied to interpreter assumptions
Mitigation: formalize semantics independent of interpreter internals.

Risk: generated code differs subtly from interpreter
Mitigation: differential testing with many small programs.

Risk: COR24 lacks enough registers for pleasant codegen
Mitigation: embrace memory-backed temporaries and tiny stack-top caching.

Risk: JIT memory management becomes dominant complexity
Mitigation: defer JIT until AOT is mature.

Risk: Pascal feature set explodes backend complexity
Mitigation: staged feature subsets and helper runtime routines.

Risk: self-hosting becomes premature constraint
Mitigation: host-first, self-host later.

If you can do that, the rest becomes an engineering expansion rather than a research mystery.

How can Rust and C share code?
Short answer
Do not try to share implementation code first. Share the specification.

Then, only share generated artifacts where that clearly helps.

Your proposed idea:

extract data structures and pseudocode from Rust to inform C

That is directionally right, but too weak if done informally.
You need something more disciplined than “look at the Rust and port it by hand.”

The right model is:

share formal schemas and docs

share test vectors

optionally generate small common headers/tables

keep actual lowering/codegen implementations separate

Best sharing strategy
Layer 1: Shared written spec
This is the most important artifact.

Create stable docs for:

p-code opcode definitions

operand encoding

procedure metadata format

frame layout

runtime helper ABI

error semantics

lowering invariants

This is the real shared codebase, conceptually.

Without this, Rust and C will drift.

Example docs
opcode-semantics.md

ir-contract.md

runtime-calls.md

cor24-native-calling-convention.md

These should be treated almost like standards documents.

Layer 2: Shared machine-readable schema
This is the second-best thing to share.

Define the opcode table and metadata in a simple declarative format such as:

JSON

YAML

TOML

or even a custom flat table file

For example:

[[opcode]]
name = "LIT"
code = 0x01
stack_in = 0
stack_out = 1
operands = ["imm24"]

[[opcode]]
name = "ADD"
code = 0x02
stack_in = 2
stack_out = 1
operands = []
Then:

Rust can read it

a small generator can emit C headers

docs can be generated from it too

This is much better than duplicating enums by hand.

Layer 3: Generated headers/tables
This is where real cross-language sharing can happen safely.

From one schema, generate:

Rust enums/constants

C enums/macros/tables

markdown opcode tables

For example:

generated/pcode_opcodes.rs

generated/pcode_opcodes.h

This is excellent for:

opcode numbers

operand kinds

helper IDs

feature flags

error codes

This is much better than trying to translate Rust structs into C mentally.

Layer 4: Shared tests and golden files
This is probably even more valuable than shared code.

Create test assets like:

p-code input

expected decoded instruction stream

expected stack-effect analysis

expected lowered IR

expected native behavior/output

Then both Rust AOT and C JIT implementations can run the same cases.

Examples:

tests/add_loop_01.pcode

tests/add_loop_01.expected-ir.json

tests/add_loop_01.expected-output.txt

This will keep both implementations aligned.

Layer 5: Optional shared IR serialization
If useful, define a serialized lowered IR format.

Then:

Rust AOT can emit IR dumps

C-side JIT tests can compare against expected lowered form

later tools can inspect/visualize the pipeline

You do not need this on day one, but it can help a lot.

What not to do
Do not try to directly share complex implementation logic across Rust and C
For example:

porting allocator-heavy Rust compiler internals into C

trying to create identical AST/IR implementations by hand

maintaining parallel hand-written opcode tables in both languages

That becomes fragile fast.

Do not use Rust as “the source” and C as “the translation”
If the C JIT matters, it needs its own proper implementation.

Use Rust as:

the first implementation

a reference for behavior

a producer of generated tables/specs/tests

But not as an informal pseudo-language for C.

Should you extract pseudocode from Rust?
Yes, but only as design documentation
That is useful for:

lowering algorithms

register allocation heuristics

stack-depth propagation

block formation

code emission patterns

For example, document:

Stack lowering rule
For opcode ADD:
- ensure TOS in v0
- ensure NOS in v1 or spill slot s0
- emit add v1, v0
- result becomes new TOS
That kind of pseudocode is valuable.

But I would not rely on prose alone. Pair it with:

machine-readable opcode schema

golden tests

ABI docs

A better sharing model than “share code”
Think of this as sharing five artifacts:

1. Semantic spec
What the opcode means.

2. ABI spec
How lowered/generated code interacts with runtime.

3. Declarative tables
Opcode IDs, operand kinds, helper signatures, flags.

4. Test corpus
Programs, traces, expected outputs.
